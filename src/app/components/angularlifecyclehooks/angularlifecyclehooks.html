Toggle the child component on and off. Each creation triggers ngOnInit and ngAfterViewInit. Removal triggers ngOnDestroy. The event log tracks every lifecycle event.

Syntax

Semantics

ngOnChanges(changes: SimpleChanges) receives a map of every input that changed. Each entry has previousValue, currentValue, and firstChange. On the very first change detection it fires before ngOnInit. Subsequent calls happen every time the parent rebinds a value.

Syntax

Semantics

ngOnInit() runs once after the first ngOnChanges. At this point all @Input values are bound and ready. Use it for initial data fetching, form setup, or any logic that depends on inputs. It does not fire again on subsequent change-detection cycles.

Syntax

Semantics

ngDoCheck() fires on every change-detection run, not just on input changes. Use it to detect mutations that Angular's default checking cannot see (e.g. pushing into an array or mutating an object property). Call detector.detectChanges() manually to flag the change. Avoid heavy work here — it runs extremely often.

Syntax

Semantics

ngAfterContentInit() runs once after Angular projects external content into the component via <ng-content>. This is the earliest point where you can query projected children with @ContentChildren. It fires after the component's own ngOnInit.

Syntax

Semantics

ngAfterViewInit() runs once after Angular has fully initialised the component's view. All child components and @ViewChild / @ViewChildren references are available. Use it to interact with the DOM: focus inputs, start animations, or read element dimensions. If you need to make additional change-detection runs, call changeDetectorRef.detectChanges().

Syntax

Semantics

ngOnDestroy() runs once right before Angular removes the component from the DOM. Always unsubscribe from Observables, clear timers, detach event listeners, and release any held resources here to prevent memory leaks. Components wrapped in *ngIf or routed views will trigger this when destroyed.

Syntax

Semantics

Angular runs hooks in a deterministic order: Create → ngOnChanges → ngOnInit → ngDoCheck → ngAfterContentInit → ngAfterContentChecked → ngAfterViewInit → ngAfterViewChecked. On subsequent change-detection rounds only the DoCheck and After*Checked hooks fire. ngOnDestroy fires once when the component is destroyed.
